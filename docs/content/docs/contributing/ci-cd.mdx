## Conventional commits

All commit messages must use [Conventional Commits](https://www.conventionalcommits.org/) prefixes:

| Prefix | Purpose |
|---|---|
| `feat:` | New feature |
| `fix:` | Bug fix |
| `refactor:` | Code restructuring without behavior change |
| `docs:` | Documentation changes |
| `test:` | Adding or updating tests |
| `ci:` | CI/CD pipeline changes |
| `chore:` | Maintenance tasks |

Changelogs are auto-generated by [git-cliff](https://git-cliff.org/) from these prefixes.

## Versioning

Git tags are the sole version source -- there is no `VERSION` file. All components derive their version from `git describe --tags` at build time. Local development defaults to `0.0.0-dev`.

### Tag scheme

Tags follow semantic versioning with the `v` prefix:

- Stable: `v1.0.0`, `v1.2.3`
- Pre-release channels: `v0.2.0-alpha.1`, `v1.0.0-beta.2`, `v1.0.0-rc.1`

### Version exposure

| Component | How version is exposed |
|---|---|
| ControlPlane | `GET /api/v1/config` (unauthenticated), `/healthz/detail` (authenticated) |
| Agent | Reports agent version and k8s API version in `AgentHello` handshake |
| CLI | `clustral version` shows CLI + ControlPlane versions |

The ControlPlane logs a warning when an agent's version differs from its own. Version mismatches are visible in cluster list tables.

## Changelog

The `CHANGELOG.md` file is auto-generated by git-cliff on tag push. CI commits it back to `main`. Do not edit it manually.

## Docker tags

Docker images follow this tagging strategy:

| Release type | Tags applied |
|---|---|
| Stable release (e.g., `v1.2.3`) | `1.2.3`, `1.2`, `1`, `latest` |
| Pre-release alpha | `0.2.0-alpha.1`, `alpha` (floating) |
| Pre-release beta | `1.0.0-beta.2`, `beta` (floating) |
| Pre-release RC | `1.0.0-rc.1`, `rc` (floating) |

The `latest` tag is only applied to stable releases.

## CI path filtering

The CI pipeline uses `dorny/paths-filter` to detect which components changed. Only affected components are built and tested, keeping CI fast for focused changes.

## Test reporting

- `.trx` test results are uploaded to GitHub Checks via `dorny/test-reporter`
- Code coverage artifacts are uploaded for both .NET and Go

## Pull request guidelines

- Keep PRs focused on a single change
- When a change spans multiple layers (e.g., proto contract + migration + implementation), call out each layer in the PR description
- Security-sensitive changes (token handling, tunnel auth) should be flagged for explicit review
