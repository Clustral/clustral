# Clustral.ControlPlane ‚Äî Claude Code Guide

ASP.NET Core web host that is the heart of the Clustral platform. It exposes:

- **REST API** on `:5100` ‚Äî consumed by the Web UI and CLI (proxied by nginx :443).
- **gRPC mTLS server** on `:5443` ‚Äî consumed by Agents (tunnel, mTLS + JWT auth). Direct connection, no nginx.
- **Swagger UI** at `http://localhost:5100/swagger` (dev only).
- **Health endpoints**: `GET /healthz` (liveness), `GET /healthz/ready` (readiness ‚Äî MongoDB + OIDC), `GET /healthz/detail` (detailed, auth required).

## Key design decisions

### Two ports, one process
REST traffic goes to `:5100` (HTTP/1.1, proxied by nginx :443); gRPC mTLS
traffic goes to `:5443` (HTTP/2, direct ‚Äî no nginx). The split prevents the
grpc-gateway Content-Type mismatch and ensures persistent agent tunnels are
not interrupted by nginx restarts. Agents connect directly to Kestrel :5443
with mTLS client certificates + RS256 JWT authorization.

### Token storage
Raw bearer tokens are **never stored**.  Only the SHA-256 hex digest
(`TokenHash`) is persisted in the `access_tokens` table.  The hash is computed
in `AuthController.HashToken` and `AuthServiceImpl.HashToken` ‚Äî keep these
in sync.  Lookup is O(1) via the unique index `ix_access_tokens_token_hash`.

### TunnelSessionManager
A `Singleton` that maps `Guid clusterId ‚Üí TunnelSession`.  The session wraps
the live gRPC `IServerStreamWriter<TunnelServerMessage>` and a
`ConcurrentDictionary` of pending HTTP request completions.  The kubectl proxy
handler (to be built) will call `TunnelSessionManager.GetSession(clusterId)` to
forward traffic.

### OIDC Provider
JwtBearer middleware is configured with the OIDC provider authority (Keycloak,
Auth0, Okta, etc.).  The middleware fetches the JWKS automatically at startup
via OIDC discovery.  In dev, `RequireHttpsMetadata: false` allows plain-HTTP
providers.  Configuration is read from the `Oidc` section in appsettings.json
(falls back to `Keycloak` for backward compatibility).

‚öÝÔ∏è  `AuthServiceImpl.IssueKubeconfigCredential` currently has a TODO for
validating the inbound `OidcAccessToken` field.  Until that is implemented,
the gRPC method accepts any non-empty string.  Implement full validation before
exposing the port publicly.

### Startup Config Validation
`StartupConfigValidator` is a hosted service that runs first on startup.  It
validates `OidcOptions` and `MongoDbOptions` using FluentValidation, logs a
config summary box, and performs non-blocking connectivity checks (MongoDB ping
and OIDC discovery fetch) with 5-second timeouts.  Validation failures abort
startup; connectivity failures are logged as warnings only.

### EF Core conventions
- Table and column names use `snake_case` via `UseSnakeCaseNamingConvention()`.
- `Labels` on `Cluster` is stored as `jsonb` in PostgreSQL; Npgsql serialises
  `Dictionary<string, string>` natively.
- Enum columns use `HasConversion<string>()` so values are human-readable in
  the database.

## Domain-Driven Design

The ControlPlane uses pragmatic DDD (Phase 1 of a multi-phase roadmap).

### Aggregate Roots
- **AccessRequest** ‚Äî encapsulates the full JIT access lifecycle with
  `Approve()`, `Deny()`, `Revoke()`, `Expire()` methods that enforce
  state machine invariants and return `Result`. Handlers call these methods
  instead of manipulating state directly.
- **Cluster** ‚Äî aggregate root for cluster registration (methods TBD in Phase 2).

### Domain Services (`Domain/Services/`)
- **UserSyncService** ‚Äî centralizes OIDC user upsert logic previously
  duplicated in `UserSyncFilter` and `IssueKubeconfigCredential`.

### Proxy Domain Services (`Domain/Services/`)
- **ProxyAuthService** ‚Äî validates bearer tokens against `IAccessTokenRepository`,
  checks credential validity, cluster scope, and kind.
- **ImpersonationResolver** ‚Äî resolves user + groups for k8s impersonation.
  Uses `AccessSpecifications` (static assignment ‚Üí JIT grant fallback).

### Proxy Configuration (`Infrastructure/ProxyOptions.cs`)
- `TunnelTimeout` ‚Äî max wait for agent response (default 2 min)
- `RateLimiting.Enabled` ‚Äî toggle per-credential rate limiting
- `RateLimiting.BurstSize` ‚Äî token bucket capacity (default 200)
- `RateLimiting.RequestsPerSecond` ‚Äî sustained refill rate (default 100 QPS)
- `RateLimiting.QueueSize` ‚Äî queued requests before 429 (default 50)

Validated with `ProxyOptionsValidator` at startup. Defaults match k8s client-go.

### DDD Rules
- **Business logic belongs in the domain** ‚Äî aggregate methods enforce
  invariants. Handlers are thin orchestrators (fetch ‚Üí call domain method ‚Üí
  persist ‚Üí return).
- **State transitions via aggregate methods** ‚Äî never set `Status` directly
  in a handler. Call `request.Approve(...)` instead.
- **Static factories for creation** ‚Äî use `AccessRequest.Create(...)` instead
  of `new AccessRequest { ... }`.
- **Domain services for cross-aggregate operations** ‚Äî when logic spans
  multiple aggregates or external concerns (e.g., OIDC claims).

### Future Phases
- Phase 2: Domain events (AccessRequestApproved, CredentialIssued)
- Phase 3: Specifications for reusable query predicates
- Phase 4: Repository interfaces (thin wrappers over ClustralDb)

## Result Pattern

Controllers use `Result<T>` from `Clustral.Sdk.Results` to represent success or
failure without throwing exceptions on expected error paths.  Error cases are
constructed via `ResultErrors.*` factory methods (e.g. `ResultErrors.NotFound`,
`ResultErrors.Forbidden`).

The extension method `ToActionResult()` maps a `Result<T>` to the appropriate
HTTP status code (200, 400, 404, 403, etc.).  Controllers should return
`result.ToActionResult()` rather than manually constructing `ObjectResult`.

`GlobalExceptionHandlerMiddleware` catches any unhandled
`ResultFailureException` (thrown when `.Value` is accessed on a failed result)
and returns an RFC 7807 Problem Details response, ensuring consistent error
formatting across the API.

## Common tasks

### Add a migration

```bash
dotnet ef migrations add <MigrationName> --project src/Clustral.ControlPlane
```

Always review the generated migration before applying.

### Apply migrations locally

```bash
dotnet ef database update --project src/Clustral.ControlPlane
```

In `Development` mode, `Program.cs` calls `MigrateAsync()` on startup as a
convenience.  Do not rely on this in production.

### Generate an idempotent SQL script for production

```bash
dotnet ef migrations script --idempotent \
  --project src/Clustral.ControlPlane \
  -o infra/migrations/latest.sql
```

### Run the server locally

```bash
docker-compose up -d          # Start postgres + keycloak
dotnet run --project src/Clustral.ControlPlane
```

## Adding a new feature (vertical slicing + CQS)

The ControlPlane uses vertical slicing with CQS (Command-Query Separation).
Commands (writes) and queries (reads) live in separate sub-folders under each feature:

```
Features/
‚îú‚îÄ‚îÄ Shared/              ‚Üê ICommand/IQuery markers, ValidationBehavior, TokenHashingService
‚îú‚îÄ‚îÄ Clusters/
‚îÇ   ‚îú‚îÄ‚îÄ Commands/        ‚Üê RegisterCluster.cs, DeleteCluster.cs
‚îÇ   ‚îú‚îÄ‚îÄ Queries/         ‚Üê ListClusters.cs, GetCluster.cs
‚îÇ   ‚îî‚îÄ‚îÄ ClusterEventHandlers.cs
‚îú‚îÄ‚îÄ Roles/
‚îÇ   ‚îú‚îÄ‚îÄ Commands/        ‚Üê CreateRole.cs, UpdateRole.cs, DeleteRole.cs
‚îÇ   ‚îú‚îÄ‚îÄ Queries/         ‚Üê ListRoles.cs
‚îÇ   ‚îî‚îÄ‚îÄ RoleEventHandlers.cs
‚îú‚îÄ‚îÄ Users/
‚îÇ   ‚îú‚îÄ‚îÄ Commands/        ‚Üê AssignRole.cs, RemoveAssignment.cs
‚îÇ   ‚îú‚îÄ‚îÄ Queries/         ‚Üê GetCurrentUser.cs, ListUsers.cs, GetUserAssignments.cs
‚îÇ   ‚îî‚îÄ‚îÄ UserEventHandlers.cs
‚îú‚îÄ‚îÄ Auth/
‚îÇ   ‚îú‚îÄ‚îÄ Commands/        ‚Üê IssueKubeconfigCredential.cs, RevokeCredential.cs, RevokeByToken.cs
‚îÇ   ‚îî‚îÄ‚îÄ CredentialEventHandlers.cs
‚îî‚îÄ‚îÄ AccessRequests/
    ‚îú‚îÄ‚îÄ Commands/        ‚Üê CreateAccessRequest.cs, Approve, Deny, Revoke
    ‚îú‚îÄ‚îÄ Queries/         ‚Üê ListAccessRequests.cs, GetAccessRequest.cs
    ‚îú‚îÄ‚îÄ AccessRequestEnricher.cs
    ‚îî‚îÄ‚îÄ AccessRequestEventHandlers.cs
```

To add a new feature:

1. Create `Features/<FeatureName>/Commands/` and/or `Queries/` folders.
2. Define a command record implementing `ICommand<Result<T>>` or `ICommand`,
   or a query record implementing `IQuery<Result<T>>`.
3. Implement a handler class implementing `IRequestHandler<TCommand, Result<T>>`.
4. Add a `FluentValidation` validator for commands (e.g., `CreateFooValidator : AbstractValidator<CreateFooCommand>`). Queries skip validation.
5. Add a thin controller action in `Api/Controllers/` that calls `mediator.Send(...)` and maps via `ToActionResult()`. No business logic in the controller.
6. Add request/response DTOs to `Api/Models/` if not already present.
7. **Write unit tests** for the validator and handler in `Tests/Features/<FeatureName>/`.
8. **Write integration tests** in `Tests/Features/<FeatureName>/` or `Tests/Integration/` using `ClustralWebApplicationFactory`.
9. **Use FluentAssertions** in all tests (`.Should().Be(...)`) ‚Äî do not use `Assert.Equal` or `Assert.True`.
10. **Use FluentValidation** for all input validation ‚Äî do NOT use `[Required]` or other data annotations. FluentValidation handles validation via the `ValidationBehavior` MediatR pipeline.

## Adding a new gRPC method

1. Add the RPC declaration to the relevant `.proto` in `packages/proto/`.
2. Rebuild `Clustral.Sdk` to regenerate stubs.
3. Implement the method in the corresponding `*Impl.cs` file here.
4. If the method requires a new DB query, use repository interfaces or
   `ClustralDb` directly for complex cross-entity queries.
5. **Write gRPC integration tests** in `Tests/Integration/Grpc*Tests.cs` using
   `Grpc.Net.Client` against `WebApplicationFactory`. Use the `ResponseVersionHandler`
   pattern (see `GrpcClusterServiceTests.cs`) to bridge HTTP/1.1 test server with
   gRPC's HTTP/2 requirement.

## Security-sensitive paths

The following must be reviewed carefully before any change:

| Location | Risk |
|---|---|
| `AuthServiceImpl.ValidateKubeconfigCredential` | Every kubectl request goes through here; a bug here breaks all proxy auth |
| `AuthServiceImpl.IssueAgentCredential` | Bootstrap token is single-use; verify consumption logic is race-free |
| `AuthController.HashToken` | Must stay consistent with `AuthServiceImpl.HashToken` |
| `TunnelServiceImpl.OpenTunnel` ‚Äî auth block | Agent impersonation if bypassed |
